Strategies Implemented

1

🔀 Horizontal Scaling (Sharding)

Transactions are routed to 2 Postgres shards using user_id % num_shards. This distributes write load evenly across database instances, preventing any single node from becoming a bottleneck.

2 DB Shards
2

🔴 Caching (Read-Aside)

Account balances are cached in Redis. Balance checks hit Redis first (<0.5ms). On cache miss, the system falls through to the correct DB shard and populates Redis for next time.

<0.5ms Cache Hit
3

📦 Write Batching

Instead of committing each transaction individually, the batcher accumulates 100 transactions (or waits 50ms) before flushing them in a single DB transaction. This reduces IOPS by ~60%.

100 txs / 50ms flush
4

📬 Async Processing (Queues)

The API immediately returns 202 Accepted. Transactions are placed into a Go channel queue and processed by a pool of background workers at their own pace.

Decoupled API → DB
5

⚙️ Concurrency & Parallelism

Go goroutines handle multiple requests simultaneously across all CPU cores. The worker pool uses NumCPU × 2 goroutines for true parallel processing.

8 Workers on 4 Cores
6

🗂️ Database Indexing

Composite indexes on (from_account_id, created_at) enable rapid lookups for balance checks and audit queries without full table scans.

Composite Indexes
7

⚖️ Throughput vs. Latency

The benchmark clearly shows the trade-off: as concurrency increases from 10→500 workers, TPS rises 26% but P99 latency rises from 9ms → 255ms. Batching trades individual latency for aggregate throughput.

P99: 9ms → 255ms
